Vas a realizar lo que esta en "FRAMEWORK.yml", el proyecto va a estar en un archivo llamado "auth.py" y otro archivo "auth.js", los cuales se van a importar en
un archivo llamado "main.py" el cual va a absorver las apis de "auth.py" y las funciones para encriptar, y a su vez "main.py" va a tener una api "/testLess" el cual va a usar
la funcion encriptar/desencriptar de stateless, y, un "/testFull" el cual va usar las funciones de statefull, y tener bien claro que dice que se pueden usar en conjunto EN UN SISTEMA,
en una api se usa uno o el otro STATEf/l. por otro lado va a haber un "index.html" el cual importa un "scripts.js", y el "scripts.js" importa el "auth.js" con el cual va a usar las funciones para hacer
fetch a esos dos endpoints del "main.py", ademas va a incorporar y usar las funciones de auth para loguearse,registrarse,desloguearse, refrescar token.
Como reglas:
- todas las variables/constantes van a tener nombres significativos a lo que son/hacen.
- se va a buscar codigo legible y corto.
- vas a poner un comentario de que hace cada funcion antes de ella.
- el sistema es un FRAMEWORK, por lo que NADA de logica de auth tiene que estar en main.py, mas que funciones que te hagan todo el cifrado y decifrado. Tampoco ningun modelo o api de auth tiene que estar en main, porque repetimos, es un FRAMEWORK o libreria.

Como entregar:
Vas a entregar los 4 archivos dentro de un .zip, ademas vas a poner un resumen de cada archivo.

Recordatorios:

Es un FRAMEWORK, main.py solo va a importar cosas de auth.py y va a usar para /testLess y /testFull las funciones al inicio y al final para cifrar o decifrar.
Lo mismo para en scripts.js con auth.js.
Que quede claro que toda logica de auth en main o scripts los cuales son para logica de negocio esta EXTREMADAMENTE MAL.




Prompt backend:

Vas a realizar la parte BACKEND del framework definido en "FRAMEWORK.yml".
El proyecto tendrá dos archivos: "auth.py" y "main.py".

auth.py contendrá toda la lógica del framework:

funciones para cifrado y descifrado tanto stateless como statefull (cifradoStateLess, descifradoStateLess, cifradoStateFull, descifradoStateFull).

funciones de autenticación register, login, refresh, logout.

gestión de sesiones y claves (AES, curva elíptica, RSA).

manejo de Redis y PostgreSQL.

inicialización automática de claves si no existen.

funciones utilitarias como desencriptar(), encriptar(), validar(), decifrarKey(), crearKeyUser().
Debe tener comentarios claros antes de cada función explicando qué hace.
Todo el código debe tener nombres de variables significativos, ser legible y corto.

main.py será el archivo que usa el framework:

importa auth.py.

expone dos endpoints /testLess y /testFull.

/testLess usará las funciones de cifrado y descifrado de modo stateless.

/testFull usará las funciones de cifrado y descifrado de modo statefull.

no debe tener ninguna lógica de autenticación ni modelos, solo utilizar las funciones provistas por auth.py.

solo maneja el flujo de negocio y testing, no autenticación.

Al entregar, incluí los dos archivos dentro de un .zip y agregá un resumen de qué hace cada uno.



Prompt frontend:

Vas a realizar la parte FRONTEND del framework definido en "FRAMEWORK.yml".
El proyecto tendrá dos archivos: "auth.js" y "scripts.js", y además un "index.html".

auth.js contendrá toda la lógica del framework frontend:

funciones de cifrado y descifrado para stateless y statefull (cifradoStateLess, descifradoStateLess, cifradoStateFull, descifradoStateFull).

funciones de autenticación registerCall, loginCall, logout, refreshToken.

funciones de configuración (config(publicKey, urlServer)).

todas las variables deben tener nombres descriptivos y el código ser claro y corto.

incluir comentarios antes de cada función explicando qué hace.

scripts.js importará auth.js y solo usará sus funciones:

manejará las llamadas fetch hacia /testLess y /testFull del backend.

usará las funciones de auth.js para cifrar y descifrar los payloads antes y después del fetch.

usará las funciones de auth.js para loguear, registrar, desloguear y refrescar tokens.

scripts.js no debe tener ninguna lógica de autenticación interna, solo lógica de negocio y uso del framework.

index.html importará scripts.js y servirá como interfaz de prueba para las funciones del framework.

Al entregar, incluí los tres archivos dentro de un .zip y agregá un resumen de qué hace cada uno.
